# Blueprint ATS+HR — ארכיטקטורת היציבות החדשה

מסמך זה מתאר את הארכיטקטורה החדשה של "המוח" — הצנרת שקולטת את הדיווחים היומיים מהמייל, מעבדת אותם, לומדת מהם, ומציגה אותם בצורה ברורה למנכ"ל ולצוות.

## למה היה צריך לתכנן מחדש

המערכת הקודמת סבלה מארבע בעיות שורש:

1. **קבצי ה-PDF של הדיווחים היומיים מעולם לא נקראו.** קליטת המייל שמרה רק את שם הקובץ; התוכן עצמו (הדוח האמיתי) הלך לאיבוד, וה-AI ניסה לחלץ מידע מגוף המייל הריק ("Please find attached...").
2. **כשל אילם בכל שלב.** אילוץ בבסיס הנתונים דחה בשקט כל דוח שמקורו במייל; ה-Cron הריץ 7 משימות בבקשה אחת ובלע כשלים; דוחות נתקעו ב"תור" לנצח (מגבלת 5 ליום); אף אחד לא ידע ששום דבר לא עובד.
3. **לולאת הלמידה הייתה מנותקת.** ה-AI ייצר שאלות הבהרה ("מה זה PYT?") אבל הן נזרקו במקום להישמר; טבלת הידע נשארה ריקה; המוח לא יכול היה להשתפר.
4. **הפלט היה בלתי קריא.** עמוד הדוחות הציג שורות סטטוס טכניות במקום את תוכן הדיווחים.

## הארכיטקטורה החדשה

### 1. צנרת קליטה מאוחדת (`lib/operations/report-intake.ts`)

מסלול עיבוד **אחד** לכל דוח, מכל מקור:

```
מייל (PDF/DOCX מצורף) ─┐
העלאה ידנית (PDF/טקסט) ─┼─→ report-intake ─→ חילוץ טקסט (pdf-parse → Claude Vision)
WhatsApp (טקסט/תמונה) ──┘        │
                                  ├─→ חילוץ פריטים עם הקשר נלמד (extract-report)
                                  ├─→ שיוך פרויקטים/עובדים + יצירת פרויקטים חדשים
                                  ├─→ שמירת שאלות למידה → op_context_questions
                                  └─→ עדכון סטטוס + שגיאות אמיתיות ב-op_reports
```

- קליטת המייל (`/api/hr/email/ingest`) **מורידה ומפענחת את הקובץ המצורף** (PDF דרך pdf-parse, סריקות דרך Claude Vision, DOCX דרך mammoth).
- מייל מ-HR עם נושא "Consolidated/Daily Report" וקובץ מצורף מנותב **תמיד** כדוח יומי, גם אם המסווג טעה.
- ה-PDF נשמר ב-Storage כך שתמיד אפשר לעבד מחדש.

### 2. משימות רקע מפוצלות ומתועדות (`vercel.json` + `/api/cron/*`)

| Cron | תזמון (UTC) | תפקיד |
|------|------------|--------|
| `/api/cron/ingest` | 05:00, 11:00, 17:00 | קריאת מיילים + עיבוד תור הדוחות |
| `/api/cron/analyze` | 05:40 | התראות, דדליינים, תבניות חוזרות, **AI Brain** |
| `/api/cron/digest` | 06:10 | שליחת דיגסט יומי למנכ"ל |
| `/api/cron/sweep` | כל 6 שעות | רשת ביטחון: שחרור פריטים תקועים + ניסיונות חוזרים |

- **כל ריצה נרשמת** בטבלת `system_runs` (התחלה, סיום, סטטוס, שגיאה, פרטים).
- **כשל שולח התראה** — מייל בעברית למנכ"ל + רשומת `op_alerts` שמופיעה במערכת.
- דיגסט שלא הגיע לאף ערוץ נחשב כשל (לא עוד "הצלחה" כשהמנכ"ל לא קיבל כלום).
- `/api/cron/daily` נשאר כהפעלה ידנית של הכל ("הרץ הכל עכשיו").

### 3. לולאת הלמידה סגורה

```
דוח נקלט → ה-AI שואל שאלה ("מה זה PYT?") → נשמרת ב-op_context_questions
       → המשתמש עונה בעמוד "ידע" → נוצרת רשומת ידע (op_context_entries)
       → הידע מוזרק לסיווג המיילים ולחילוץ הדוחות הבאים → פחות שאלות, דיוק גבוה יותר
```

- השאלות נשמרות עכשיו גם מצנרת הדוחות הראשית (לא רק מטיוטות).
- הידע הנלמד מוזרק גם לסיווג מיילים (`classify-email`), לא רק לחילוץ.
- `/api/ai-brain/cron` רץ אוטומטית כל בוקר — תובנות ההנהלה מתעדכנות בלי לחיצה ידנית.

### 4. היומן היומי — התצוגה למנכ"ל (`/operations/reports`)

עמוד הדוחות הפך מ"טבלת סטטוסים" ל**יומן יומי**:

- כרטיס לכל דיווח, מקובץ לפי יום: מי שלח, סיכום AI בעברית, כמה פריטים.
- **פריטים שדורשים החלטת מנכ"ל מוצמדים למעלה** עם סימון אדום.
- שאר הפריטים מקובצים לפי פרויקט, עם תגיות סטטוס/דחיפות/דדליין/אחראי.
- דוחות שנכשלו מציגים את השגיאה האמיתית + כפתור "עבד מחדש".
- פס "מצב המערכת" (למנהל בלבד): ריצה אחרונה, כשלים ב-24 שעות, פריטים תקועים.
- הדיגסט במייל/WhatsApp מקשר ליומן החי (התצלום הסטטי נשאר כגיבוי).

### 5. תיקוני בסיס נתונים (מיגרציה 018)

- `system_runs` — יומן ריצות של כל משימות הרקע.
- `op_reports.source_type` מקבל `'email'` (האילוץ הישן דחה בשקט את כל דוחות המייל).
- `op_reports.attempts` — ספירת ניסיונות לטובת ה-sweeper (עד 3 ניסיונות).
- `op_context_questions.draft_id` אופציונלי + עמודת `report_id` — שאלות למידה מכל הצנרת.

## שחזור דוחות שאבדו

`POST /api/maintenance/backfill-email-reports?days=30` — סורק מיילים של דוחות מ-30 הימים האחרונים שלא הניבו פריטים, מוריד מחדש את הקבצים המצורפים מ-Gmail, ומכניס אותם לתור העיבוד. להרצה חד-פעמית אחרי הפריסה.

## בדיקת תקינות

- `GET /api/system/health` (מנהל) — ריצות אחרונות, כשלים, פריטים תקועים, שאלות ממתינות.
- בדיקה ידנית: `GET /api/cron/ingest` עם `Authorization: Bearer $CRON_SECRET`.

## משתני סביבה רלוונטיים

- `SYSTEM_ALERT_EMAIL` (אופציונלי) — נמען התראות הכשל; ברירת מחדל: `OPERATIONS_CEO_EMAIL`.
- `CRON_SECRET` — חובה בפרודקשן; Vercel שולח אותו אוטומטית ל-crons.
